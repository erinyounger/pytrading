# 功能规范: 测试基础设施搭建

**功能分支**: `001-test-infra`
**创建时间**: 2026-03-07
**状态**: 草稿
**输入**: 用户描述: "基于章程原则，为项目补充后端pytest和前端Jest测试基础设施"

## 用户场景与测试 *(必填)*

### 用户故事 1 - 后端开发者运行单元测试 (优先级: P1)

作为后端开发者, 我希望能通过一条命令运行所有后端测试并查看覆盖率报告, 以便在提交代码前快速验证功能正确性, 满足章程原则 III 的 80% 覆盖率要求.

**优先级原因**: 后端目前完全没有自动化测试框架, 这是建立测试文化的最基础前提. 章程原则 III 明确要求使用 pytest 且新代码覆盖率达 80%.

**独立测试**: 可以通过在项目根目录执行单一测试命令, 验证测试发现、执行、覆盖率报告生成全流程正常工作.

**验收场景**:

1. **给定** 项目已配置 pytest, **当** 开发者在项目根目录执行测试命令, **那么** 系统自动发现并运行 `tests/` 下所有测试文件, 输出测试结果和覆盖率摘要
2. **给定** 测试需要数据库操作, **当** 测试执行时, **那么** 系统使用独立的测试数据库环境, 测试结束后自动清理数据, 不影响开发或生产数据
3. **给定** 测试依赖外部服务(MyQuant/AKShare), **当** 测试执行时, **那么** 系统使用预定义的模拟数据替代真实 API 调用, 测试可在无网络环境下完成

---

### 用户故事 2 - 前端开发者运行组件测试 (优先级: P2)

作为前端开发者, 我希望能运行前端测试套件来验证组件行为和 API 调用的正确性, 以便在修改 UI 后快速确认没有引入回归问题, 满足章程原则 IV 的 70% 覆盖率要求.

**优先级原因**: 前端已安装测试依赖(Jest + RTL)但没有任何测试文件. 建立测试模式和示例对后续组件开发至关重要.

**独立测试**: 可以通过在前端目录执行测试命令, 验证组件渲染、交互事件和 API mock 全部正常工作.

**验收场景**:

1. **给定** 前端已配置测试环境, **当** 开发者执行前端测试命令, **那么** 系统自动发现并运行所有测试文件, 输出测试结果和覆盖率报告
2. **给定** 组件依赖后端 API, **当** 测试执行时, **那么** 系统使用 mock 替代真实 API 请求, 测试不依赖后端服务运行
3. **给定** 组件使用 Ant Design 和 lightweight-charts, **当** 测试执行时, **那么** 第三方库被正确模拟, 测试聚焦于业务逻辑而非第三方库内部行为

---

### 用户故事 3 - 开发者为交易策略编写参数化测试 (优先级: P3)

作为量化策略开发者, 我希望有一套策略测试模板和共享的市场数据夹具, 以便为每个策略快速编写覆盖上涨/下跌/震荡行情的参数化测试, 满足章程原则 II 的信号可复现性要求.

**优先级原因**: 策略逻辑是系统核心, 其正确性直接影响交易决策. 章程原则 II 要求信号可复现, 原则 III 要求参数化策略测试.

**独立测试**: 可以通过运行策略相关测试, 验证同一组市场数据在多次运行中产生完全一致的交易信号.

**验收场景**:

1. **给定** 测试夹具提供预定义的市场行情数据(上涨/下跌/震荡), **当** 开发者对某策略运行参数化测试, **那么** 每组行情数据独立执行, 各自验证信号输出是否符合预期
2. **给定** 相同的策略参数和市场数据, **当** 测试重复执行多次, **那么** 每次产生完全相同的测试结果(信号确定性)

---

### 边界情况

- 当测试数据库连接不可用时, 数据库相关测试应被自动跳过并给出明确提示, 而非导致整个测试套件失败
- 当外部依赖(MyQuant/AKShare)的 API 接口发生变更时, mock 数据结构应便于更新, 且更新不影响已有测试逻辑
- 当测试并发执行时, 各测试用例之间不应产生数据竞争或互相干扰
- 当前端第三方图表库(lightweight-charts)版本升级时, 快照测试应能检测到渲染变化

## 需求 *(必填)*

### 功能需求

- **FR-001**: 系统必须提供后端测试配置, 使开发者能通过单一命令发现并运行 `tests/` 目录下所有测试
- **FR-002**: 系统必须提供覆盖率收集和报告功能, 展示每个模块的行覆盖率百分比
- **FR-003**: 系统必须提供数据库测试夹具, 自动创建测试数据库会话并在测试结束后回滚所有变更
- **FR-004**: 系统必须提供外部服务(MyQuant SDK、AKShare)的 mock 工厂, 返回预定义的市场数据
- **FR-005**: 系统必须提供市场行情测试数据集, 包含上涨趋势、下跌趋势、震荡盘整三种典型行情的 K 线数据
- **FR-006**: 系统必须提供前端测试配置, 使开发者能通过单一命令运行所有前端测试并查看覆盖率
- **FR-007**: 系统必须提供前端 API 层 mock 工具, 允许测试中拦截和模拟后端 API 响应
- **FR-008**: 系统必须提供前端第三方库(Ant Design、lightweight-charts)的测试适配, 确保测试环境中组件可正常渲染
- **FR-009**: 系统必须提供至少一个后端示例测试, 演示单元测试、mock 和 fixture 的标准用法
- **FR-010**: 系统必须提供至少一个前端示例测试, 演示组件渲染、交互测试和快照测试的标准用法
- **FR-011**: 后端测试命名必须遵循 `test_<被测功能>_<场景>_<预期结果>` 格式

### 关键实体

- **测试配置**: 定义测试发现规则、覆盖率收集范围、报告格式等运行参数
- **测试夹具**: 可复用的测试前置条件, 包括数据库会话、mock 对象、市场数据等, 在测试间共享但互不干扰
- **Mock 工厂**: 生成外部服务模拟对象的工具, 封装 MyQuant/AKShare 的典型返回值, 支持按场景切换
- **市场数据集**: 预定义的标准化 K 线行情数据, 按趋势类型(上涨/下跌/震荡)分类, 用于策略参数化测试

### 假设

- 后端测试数据库使用 SQLite 内存模式替代 MySQL, 避免测试对外部数据库服务的依赖
- 前端测试使用 react-scripts 内置的 Jest 配置, 不额外引入独立的 Jest 配置文件
- 现有 `test/` 目录中的手动脚本保留不动, 新的自动化测试放在独立的 `tests/` 目录(后端)和 `frontend/src/__tests__/` 目录(前端)
- MyQuant SDK mock 数据基于真实 API 返回值结构, 但使用固定的合成数据确保可复现性

## 成功标准 *(必填)*

### 可衡量的结果

- **SC-001**: 开发者能在 30 秒内完成全部后端测试套件的执行(含覆盖率报告生成)
- **SC-002**: 开发者能在 30 秒内完成全部前端测试套件的执行(含覆盖率报告生成)
- **SC-003**: 后端示例测试覆盖率达到 80% 以上(针对被测模块)
- **SC-004**: 前端示例测试覆盖率达到 70% 以上(针对被测组件)
- **SC-005**: 所有测试可在无网络、无外部数据库的隔离环境中成功运行
- **SC-006**: 新开发者能在 10 分钟内理解测试结构并参照示例编写第一个测试用例
